# 蓝牙按键 · 版本变更记录

## v3.15（versionCode 18）— 车外喊话：消除回声 + 尽量提响度

### 背景
- v3.8 的车外喊话用软件回环 `AudioRecord(MIC) → AudioTrack(OCC_MIC)`，录音源是原始 `MIC`，无任何回声消除，实测有「很难听的回声」。目标：跟原车车外声音一样——自带降噪、无回声。

### 排查过程（记录踩过的坑）
- **v3.10 只调 `Audio.setMicOccupyState`** → 无声。反编译 `AudioImpl` 证实该接口（及 `setOccupyVolumeLevel`/`setMicInputAmplitude`）**只向 HMI 仪表推送 JSON 通知点亮「喊话中」图标，不参与音频路由**。
- **v3.11 用 `AudioManager.setParameters("extspkr_switch=open")`** → 无声。全系统无人写该标志，它是**只读状态**，非开关。
- **结论**：能出声的路由只有 v3.8 的 **OCC_MIC**（`Policy.getAudioAttributes(USAGE_OCC_MIC/CONTENT_TYPE_OCC)` → `AudioTrack` → `TERT_TDM_RX` → TFA 车外功放）。回声的唯一病因是录音源用了原始 `MIC`。

### 改动（最终方案）
- `audio/ExteriorVoice` 重写：恢复 OCC_MIC 输出路由；录音源改为 **`MediaRecorder.AudioSource.VOICE_COMMUNICATION`**（平台为「扬声器+麦克风同时工作」准备的音源，启用 DSP 自带 AEC/降噪/AGC），并在录音会话显式挂 `AcousticEchoCanceler` + `NoiseSuppressor` 双保险。**不挂 `AutomaticGainControl`**——AGC 会把喊话拉平变小声。
- 响度：软件增益 **10 倍 + `tanh` 平滑饱和限幅**（语音是尖峰信号，抬高平均电平把小功率车外喇叭催到上限，饱和只加谐波、不产生回声/硬削顶破音）；同时把 OCC 音量流与 `AudioTrack` 音量拉满。
- `toggle()/setOn()/isOn()` 对外签名不变，`ActionExecutor`、设置页调用无需改。
- 设置页「车外喊话」副文案改为「麦克风实时喊向车外喇叭 · DSP 自带降噪回声消除 · 不保存、不上传」。

### 已知限制
- 全车对外发声仅一条 OCC 通路、一个小功率 TFA 车外喇叭（行人提示/AVAS 用）。锁车音效是功放固件自带的标定音调、绕过安卓满功率播出，无法把任意语音塞进那条路径。喊话响度已顶到 OCC 数字通路上限，再提升需改功放 ACDB 标定（有推爆喇叭风险，未做）。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.15.apk`（versionCode 18）。

## v3.9（versionCode 12）— 按键诊断（看按键为什么没反应）

### 背景
- 按键没反应通常卡在三步之一：没收到广播 / 收到但解密失败（BindKey 错）/ 解密成功但 objId·value 没匹配任何手势（广播码不对）。抓包助手只解码不给「裁决」，本版补一个诊断页，直接指出卡在哪一步。

### 改动
- `AppLog` 新增「诊断模式」：`setDiagEnabled/isDiagEnabled` + `diag(tag,msg)`。仅诊断页打开时放行逐帧裁决日志，关闭零开销（不写 logcat、不分发）。
- `ButtonService.onFrame`：诊断模式下对每台已接入设备逐帧打印来源、rssi、解密出的 objId·value，并在每个提前返回点给出裁决（✗找不到产品适配 / ✗未配置 BindKey / ✗未匹配手势 / ·去重忽略 / △命中未绑动作 / ✓命中并执行）。非诊断模式行为不变。
- 新增 `ui/DiagActivity` + `activity_diag.xml`：只读日志页，进入即开诊断模式并订阅 `AppLog`，离开即关；确保常驻服务在跑。
- 设置页新增入口「按键诊断 · 看按键为什么没反应」；Manifest 注册 `DiagActivity`。

### 用法
- 设置 → 按键诊断 → 按一下实体按键：有 ✓ 即正常；出现 ✗解密失败=BindKey 错；✗未匹配=广播码不在映射（可自学习）；完全没有新行=没收到广播 / MAC 不符（去抓包助手核对）。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.9.apk`（versionCode 12）。

## v3.8（versionCode 11）— 车外喊话（麦克风实时喊向车外喇叭）

### 背景
- 新增一路系统能力：把车内麦克风的实时音频喂到车机的「车外喇叭 / 行人提示喇叭（PA）」，实现「实时喊话」。可用蓝牙按键一按开、再按关，也可在设置页手动开关。
- 实现完全独立自研，不依赖任何外部 Demo APK。

### 路由原理
- 车机 ecarx `Policy`（`com.ecarx.xui.adaptapi.policy`）提供按名解析的车外音频属性：usage=`USAGE_OCC_MIC`、contentType=`CONTENT_TYPE_OCC`。用它构造 `AudioTrack` 的 `AudioAttributes`，车机音频策略即把这路 PCM 输出到车外喇叭，无需单独切 AVAS 开关。
- 数据链：`AudioRecord`（MIC，16kHz 单声道 PCM16）实时读取 → 直接 `AudioTrack.write` 到车外喇叭。全程只在内存中转，不落盘、不上传。

### 改动
- 新增 `audio/ExteriorVoice`：进程内单例引擎，`toggle()/setOn()/isOn()`；后台线程跑录音→放音循环，关闭即停并释放。非 ecarx 环境静默降级。
- `ActionCatalog` 新增系统动作 `sys_exterior_ptt`「车外喊话（切换）」，任意蓝牙手势均可绑定。
- `ActionExecutor.executeSystem`：命中 `sys_exterior_ptt` 调 `ExteriorVoice.toggle`。
- 设置页新增「车外喊话」手动开关（`activity_settings.xml` + `SettingsActivity`），开启时按需申请 `RECORD_AUDIO`。
- `AndroidManifest` 新增 `RECORD_AUDIO`、`MODIFY_AUDIO_ROUTING`、`MODIFY_AUDIO_SETTINGS` 权限。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.8.apk`（versionCode 11）。

## v3.7（versionCode 10）— 自学习与内置手势并存

### 背景
- v3.6 里设备一旦有任一自学习事件，`hasLearned()` 即让运行时只走 `matchLearned`、编排页只显示已学动作，导致内置的单击/双击/长按被整体停用。本版改为二者并存。

### 改动
- `ButtonService.onFrame`：自学习事件优先匹配（`matchLearned`），未命中再回落到 Profile 的 `adapter.parse`；因命中即返回，不会同一帧双触发。
- `DeviceEditorActivity.refreshGestures`：Profile 内置手势 + 自学习事件合并显示（学左旋后，单击/双击/长按仍在）。
- 通用载体 `generic.mibeacon` 自身只有占位手势，不并入（`GENERIC_CARRIER_ID` 过滤），避免出现永不触发的占位行。

### 说明
- 若把某内置手势对应的 objId 重新自学习一遍，则该 objId 由自学习事件接管（自学习优先），内置同 objId 手势对其失效——属预期，勿重复学内置手势。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.7.apk`（versionCode 10）。

## v3.6（versionCode 9）— 内置自学习接入（现场教一遍即用）

### 背景
- v3.5 的抓包助手仍是「抓完人肉写 Profile」。本版把「抓包 → 生成规则 → 直接接入使用」整条闭环搬进 App：对同一动作连续操作 3 次，App 自动解密并生成事件签名，命名后即可绑定动作，无需改代码、无需预置 objId 映射。
- 直接解决旋钮「左旋/右旋」这类同 objId、只靠 value 字节区分的场景。

### 自学习数据模型
- 新增 `LearnedEvent`（automation）：一条学到的按键签名 = `objId` + value `mask`/`expected`。`matches(objId, value)` 要求 objId 相等且每个掩码位 `(value[i]&mask[i])==(expected[i]&mask[i])`；带 JSON 序列化。
- `BoundDevice` 新增 `learned` 列表 + `hasLearned()`；随设备存进加密存储（`toJson`/`fromJson` 读写 `learned` 数组）。
- `MiBeacon.Result` 的 `value` 字段（v3.5 已加）在此复用。

### 学习流程（编排页 → 自学习手势 → 学习新动作）
- 新页 `LearnActivity`：独立 `BleScanner` 只读监听目标 MAC 的 FE95，解密后按 `frameCounter` 去重，对同一动作采样 `NEEDED=3` 次。
- 多帧求稳定字节：同位全等 → 判别位 `0xFF`，有变化 → 忽略 `0x00`；`expected` 取首样本。据此自动区分左右旋等。
- 采样一致性校验（objId 必须相同）+ 与已学事件的双向冲突检测（无法区分则拒绝并提示换动作）；命名后 `RuleStore.upsert` + `ButtonService.reload`，即时生效。
- 支持删除已学动作（连带移除其手势绑定）。

### 运行时与编排
- `ButtonService`：设备 `hasLearned()` 时走 `matchLearned`（解密 FE95 → 按 `LearnedEvent.matches` 命中 → 以事件 id 作 gestureId），否则沿用 Profile 的 `adapter.parse`。
- `DeviceEditorActivity`：有自学习事件时以其为手势来源（`Gesture(e.id, e.label)`），否则用 Profile 声明的手势；`onResume` 从存储同步最新已学事件；新增「自学习手势」入口按钮。

### 通用载体 Profile
- 新增 `assets/profiles/generic_mibeacon.json`（`generic.mibeacon`「通用米家按键（自学习）」，BINDKEY16/mibeacon）：`match` 用一条永不命中的 `namePrefix`，故不会在扫描时抢识别真实设备，但会出现在「选择设备型号」列表——全新未知米家设备可先按它接入，再自学习。

### 支持范围（如实说明）
- 支持被动广播型按键：加密米家需 BindKey（本版自学习即针对此类），明文广播理论可扩展。
- 不支持需 GATT 连接交互的设备，及没有任何稳定判别字节的设备。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.6.apk`（versionCode 9）。
- `AndroidManifest` 注册 `LearnActivity`。

## v3.5（versionCode 8）— 电量显示 + 抓包助手（补齐设备相关项）

### 背景
- v3.4 之后仍缺「按键电量、旋钮左右旋、长按连发」三项。它们都依赖设备真实广播里的取值编码，硬写等于猜。本版把能做实的做实，剩下两项用抓包助手从「猜代码」变「抓一次即可配」。

## v3.3（versionCode 6）— 修复米家拉设备返回 0

### 根因：apiPost 是「不能用」的混合实现
- 旧代码发送了 `MIOT-ENCRYPT-ALGORITHM: ENCRYPT-RC4` 头，却用 HmacSHA256 对**明文** data 签名，且签名路径含 `/app` 前缀——既不符合 RC4 加密法、也不符合旧的明文法，服务端解密/验签失败，`result.list` 为空，故「从米家没读到设备」。

### 改为对齐开源 Xiaomi-cloud-tokens-extractor 的 RC4 加密法
- 签名路径去掉 `/app` 前缀（用传入的 `path`，如 `/home/device_list`）。
- 流程：明文参数先算 `rc4_hash__`(SHA1) → 全部参数值 RC4(ARCFOUR，丢弃前 1024 字节密钥流) 加密 → 对密文再算 `signature`(SHA1) → 追加 `ssecurity`、`_nonce` → POST。
- 响应同为 RC4(base64) 密文，用相同 `signedNonce` 解密得明文 JSON。
- 删除旧的 HmacSHA256 `signature()` 与 `TreeMap`/`Mac` 引用，新增 `Cipher` RC4 helper。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.3.apk`。

## v3.2（versionCode 5）— 加设备入口重做 + 全页返回键

### 根因：加设备流程被「BLE 扫描命中已登记产品」一道卡死
- 旧逻辑：`onFrame → registry.identify()` 只认内置 Profile；未命中就一行不出，手动输入 / 米家设备列表都藏在「扫到行→点添加→弹窗」里，导致用户看到「没获取到 / 没入口 / 米家设备不显示」。
- 关键事实：运行时 `ButtonService.onFrame` **只按 MAC 命中**已接入设备（再用存储的 productId 取适配器解析），与 `identify()` 无关。故只要「真实广播 MAC + 正确产品 + 正确密钥」即可生效——这是修复的基础。

### 添加设备页重做（三条路并存，互不依赖）
- **① 米家设备列表**（常驻按钮）：登录后列出账号下全部设备（名称/MAC/有无密钥），选一个 → 选对应产品型号 → 命名落库。
- **② 手动输入 Key**（常驻按钮）：选产品 → 输入 MAC（12 hex，可带冒号）→（加密产品再输 32 位 BindKey）→ 命名。
- **③ 扫描列表**：除已识别产品外，**未识别的 FE95 米家广播也会显示**（标「未知米家设备」）；点添加 → 选产品 → 取密钥（可「从米家列表指认」把密钥配给真实广播 MAC，绕开 MAC 不一致）→ 命名。
- 预取 toast 改为「已从米家读取 N 个设备（含密钥 M 个）」，便于判断是否真的拉到数据。

### 全部二级页加返回键
- 新增 `BackBarActivity` 基类：`setContentView` 后自动把布局里 id 为 `btn_back` 的「‹ 返回」接上 `finish()`。
- 账号 / 添加设备 / 按键编排 / 动作选择 / 设置 / 共创广场 六个二级页顶部统一加返回栏。

### 构建
- 产物：`app/build/outputs/apk/release/蓝牙按键_v3.2.apk`。

## v3.0（versionCode 3）— 首个完整可用版

包名 `com.selfclink.ble`，应用名「蓝牙按键」，平台签名（CN=Android，android@android.com）。

### 设计要点
- **共创框架**：设备识别/解析由声明式 JSON 配置（DeviceProfile）驱动，两种 codec：
  - `mibeacon`：FE95 加密广播（手写 AES-CCM），需 16 字节 BindKey；objId→手势由 `gestureObjMap` 配置。
  - `advmatch`：明文广播按 offset/mask/equals 字节匹配，无需密钥。
  - 内置示例：米家蓝牙旋钮（`mijia_knob_ts00.json`）。
- **凭据策略**：米家账号票据仅存内存（MiAccount），退出即清；BindKey 属敏感数据，存于 EncryptedSharedPreferences（AES256，Keystore 密钥）。
- **每个设备不同图标**：IconPicker 按 productId 选图标、按 mac 哈希分配配色。

### 本版实现的界面
- **主页**（MainActivity）：设备网格，在线状态 + 绑定数；长按重命名/删除；「添加设备」卡；顶部账号/设置入口。
- **账号页**（AccountActivity）：账号密码 / 扫码 两种登录自由切换，含免责说明，区域可选；扫码用 zxing 渲染二维码 + 长轮询。
- **添加设备向导**（AddDeviceActivity）：被动扫描命中已登记产品的设备 → 加密设备自动从米家云取 BindKey（或手动输入 32 hex）→ 命名 → 落库。
- **按键编排页**（DeviceEditorActivity）：改设备名；按产品声明的每个手势（单击/双击/长按…）分别绑定一组动作。
- **动作选择器**（ActionPickerActivity）：按 空调/座椅/车身/其它/系统 分类多选，选中顺序即执行顺序，P 档拦截项标注。
- **共创广场**（CommunityActivity）：列出内置 + 导入配置；从文件 / 粘贴导入；系统分享导出（不含 BindKey）；删除用户配置。
- **设置页**（SettingsActivity）：共创广场、米家账号入口、关于与版本号。

### 运行机制
- 前台常驻服务（ButtonService）被动扫描 BLE（SCAN_MODE_LOW_LATENCY）→ 识别产品 → 解析手势 → 执行动作；看门狗周期重开扫描保活。
- 去重：mibeacon 用帧计数，advmatch 用 700ms 时间窗。

### 米家云对接（MiCloud）
- 算法对齐开源 Xiaomi-cloud-tokens-extractor：serviceLogin→serviceLoginAuth2→securityToken；HMAC-SHA256 签名请求拉 `device_list` 与 `blt_get_beaconkey`。
- 仅与小米官方域名通信，票据/密码不落盘、不上传第三方。
